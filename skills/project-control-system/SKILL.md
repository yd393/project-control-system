# 项目总控

## 目的

把一个正在变复杂的项目整理成可推进、可追踪、可复盘的结构。

这个 skill 不绑定任何领域。不要默认它是量化、软件、产品或研究项目；先从项目现状中提取主线、阶段、角色、决策和产物。

## 适用场景

使用本 skill，当用户需要：

```text
1. 整理项目文件夹层级。
2. 建立或修复总控文档。
3. 把混在一起的任务、决策、复盘、草稿分类。
4. 按阶段拆分后续任务。
5. 按角色分配责任和交付物。
6. 把当前主线、实验分支、归档内容分清。
7. 防止所有内容堆在一个文档里。
```

不要把本 skill 当成项目管理软件。它的目标是为文件系统和文档建立清晰秩序。

## 核心原则

```text
1. 总控文档只做导航、状态和下一步，不堆长细节。
2. 专项文档一次只解决一个主题。
3. 当前有效内容和历史归档分开。
4. 主线、实验、执行、复盘、归档分开。
5. 任务必须有阶段、负责人、状态和交付物。
6. 裁决、方案、实施、验证、复盘分开记录。
7. 先整理现状，再决定是否迁移文件或创建新文档。
8. 先确认实际项目根目录，再整理文档、目录和 Git 留档。
9. 每完成一个可解释阶段，都要考虑是否需要 Git 留档。
```

## 默认文件夹结构

优先使用这套通用结构；如果项目已有清晰惯例，贴合现有惯例调整命名。

```text
docs/
├── 00_control/      # 总控、阶段任务、角色任务、决策入口
├── 01_baseline/     # 当前主线、版本边界、稳定口径
├── 02_execution/    # 执行方案、操作手册、日志字段、运行规则
├── 03_research/     # 新假设、实验方向、候选方案、未验证想法
├── 04_review/       # 回测、复盘、结果总结、验收记录
└── 99_archive/      # 旧稿、备份、废弃路线、历史材料
```

如果用户偏好中文目录，可使用：

```text
docs/
├── 00_总控
├── 01_主线
├── 02_执行
├── 03_研究
├── 04_复盘
└── 99_归档
```

## 标准文档

最小可用文档集：

```text
docs/00_control/项目总控.md
docs/00_control/阶段任务表.md
docs/00_control/角色任务表.md
```

按需要再增加：

```text
docs/01_baseline/主线定义.md
docs/01_baseline/版本边界.md
docs/02_execution/执行方案.md
docs/02_execution/操作手册.md
docs/02_execution/日志字段.md
docs/03_research/研究计划.md
docs/04_review/复盘记录.md
```

## 工作流

### 1. 确认项目根目录

不要默认 skill 或插件所在目录就是实际项目目录。先根据以下信号确认本次要管理的项目根目录：

```text
1. 用户明确给出的项目路径。
2. 当前工作区根目录。
3. 当前打开或正在修改文件所属的 Git 根目录。
4. 已存在的 docs、src、strategies、package、配置文件或业务文档所在目录。
```

如果插件目录和实际代码目录不是同一个地方，所有项目整理、文档创建、归档和 Git 留档都必须围绕实际项目根目录进行。

如果实际项目目录不在当前可读写范围内，不要猜测或绕开权限；先说明需要访问哪个路径，再等待用户确认或授权。

### 2. 盘点现状

先读取项目中的目录、关键文档和当前打开的文件。识别：

```text
1. 当前主线是什么。
2. 哪些文件是正式产物。
3. 哪些内容是实验、草稿或历史。
4. 当前未解决问题有哪些。
5. 用户已经明确的边界是什么。
```

不要先大规模移动文件。先形成判断，再小步整理。

### 3. 建立总控

总控文档包含：

```text
1. 项目目的。
2. 当前总状态。
3. 当前主线。
4. 当前边界。
5. 文档结构入口。
6. 阶段拆分。
7. 下一步任务入口。
```

总控文档避免记录长讨论和详细方案。详细内容放专项文档，只在总控里链接。

### 4. 拆阶段任务

阶段任务表使用通用层级：

```text
L0：决策层
L1：研究 / 需求层
L2：方案设计层
L3：实现 / 执行层
L4：验证 / 复盘层
```

每个任务至少记录：

```text
任务编号
状态
负责人
需要回答的问题
交付物
下一步
```

### 5. 拆角色任务

角色不要写死为具体模型名，除非用户明确需要。优先使用通用角色：

```text
Owner / Decision Maker：最终决策和风险承担
Research Lead：方向判断、问题定义、方案裁决
Implementation Lead：实现设计、代码或执行方案
Execution Agent：文档落地、整理、检查、记录
Reviewer：风险复核、验收、反例检查
```

如果用户有自己的角色体系，沿用用户体系。

### 6. 记录裁决

当出现方案分歧或风险问题时，裁决记录必须写清：

```text
1. 问题是什么。
2. 上报依据是什么。
3. 裁决结论是什么。
4. 通过的部分。
5. 必须修正的部分。
6. 下一位角色要交付什么。
```

不要把“讨论中”伪装成“已裁决”。

### 7. 维护归档

归档原则：

```text
1. 不删除历史，只移动到 archive。
2. 废弃方案要说明废弃原因。
3. 当前有效版本必须比历史版本更容易找到。
4. 备份文件不要留在根目录或主工作区。
```

### 8. 变更留档

Git 是默认的项目留档方式。每完成一个明确、可解释的小阶段，都要检查是否应该留档：

```text
1. 建好目录结构后。
2. 建好或修复总控文档后。
3. 阶段任务表整理完成后。
4. 角色任务表整理完成后。
5. 归档迁移完成后。
6. 重要裁决、方案确认或验收记录完成后。
```

留档前必须先检查实际项目根目录的 Git 状态，区分：

```text
1. 本轮由当前工作产生的改动。
2. 用户此前已有的未提交改动。
3. 与本阶段无关的改动。
4. 新增、移动、删除和重命名的文件。
```

只提交本轮、本阶段相关的改动。不要把用户已有改动、无关改动或不确定来源的改动混入提交。

如果用户明确授权 Codex 提交，可以在阶段完成后执行 Git 留档。提交信息要简短说明本阶段成果，例如：

```text
docs: establish project control structure
docs: add phase task table
docs: archive outdated project drafts
docs: record project decision
```

如果用户没有明确授权 Codex 提交，或者用户主要在其他工具里复核改动，只输出留档提醒、改动摘要、建议检查文件和建议提交信息，不擅自提交。

不要使用会改写历史或丢失内容的 Git 操作，例如 reset、rebase、amend、clean、checkout 覆盖文件，除非用户明确要求并理解影响。

## 输出风格

用户要“整理项目”时，优先实际创建或更新文档和目录。

完成后简短说明：

```text
1. 建了哪些目录。
2. 新增或更新了哪些文档。
3. 当前下一步是什么。
4. 哪些内容没有动。
5. Git 留档状态：是否已提交、建议提交信息、哪些改动未提交及原因。
```

除非用户要求，不要输出很长的项目管理理论。

## 重要边界

```text
1. 不替用户做业务裁决，除非用户明确要求裁决。
2. 不把领域专用规则写进通用模板。
3. 不为了整洁而擅自删除资料。
4. 不把所有细节塞进总控文档。
5. 不在方案未确认前推动代码或执行层大改。
6. 不默认插件目录就是项目目录。
7. 不默认替用户提交 Git；没有明确授权时只提醒和建议。
```
